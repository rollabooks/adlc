\section*{What You Will Build}
In this chapter, you will transform the Notes API from Chapter~6 from in-memory storage into a \textbf{backend with a real database}:
\begin{itemize}
  \item PostgreSQL as a relational database
  \item Prisma as an ORM (Object-Relational Mapper)
  \item Automatic migrations for schema management
  \item First contact with MCP (Model Context Protocol) for database integration
\end{itemize}

\textbf{Estimated time}: 45--60 minutes\\
\textbf{Prerequisite}: \texttt{notes-api} project from Chapter~6

\begin{notebox}
\textbf{Theory Box --- Databases, ORMs, and Migrations.} In this chapter, you will move from a simple in-memory array to a \textbf{real relational database} (PostgreSQL). Data is organized into \textbf{tables} with typed columns (text, numbers, dates). An \textbf{ORM} (Object-Relational Mapper) like Prisma translates operations into the database language (SQL) for you: you write \texttt{prisma.note.create(\{title: "Hello"\})} and Prisma generates the corresponding SQL. \textbf{Migrations} are scripts that modify the database structure in a traceable way --- like a changelog for tables. The AI will generate all of this; your task is to verify that the data schema makes sense.
\end{notebox}

\begin{notebox}
\textbf{Tooling Box --- Stack chosen for this example.}
\begin{itemize}
  \item \textbf{Database:} PostgreSQL~16
  \item \textbf{ORM:} Prisma~6.x
  \item \textbf{Migrations:} Prisma Migrate (built in)
\end{itemize}
\textbf{Equivalent alternatives:} MySQL/MariaDB with TypeORM, Python with SQLAlchemy, Go with GORM, MongoDB with Mongoose. The \textbf{pattern} (relational database + ORM + trackable migrations) is universal. If you had chosen Drizzle instead of Prisma, or SQLAlchemy instead of both, the 0-code method (Context Engineering + ADLC) would have been identical.
\end{notebox}

\section{Install PostgreSQL}

\begin{praticabox}[title={Hands-On --- PostgreSQL Setup}]
\textbf{Option A --- Docker (recommended):}
\begin{lstlisting}[language=bash]
docker run --name notes-db \
  -e POSTGRES_USER=notes \
  -e POSTGRES_PASSWORD=notes123 \
  -e POSTGRES_DB=notesdb \
  -p 5432:5432 -d postgres:16
\end{lstlisting}

\textbf{Option B --- Local installation (Windows):}
Download from \url{https://www.postgresql.org/download/windows/}, install with pgAdmin, create a database called \texttt{notesdb}.

\textbf{Option C --- Cloud (Supabase/Neon):}
Create a free project on \url{https://supabase.com} or \url{https://neon.tech} and copy the connection string.

\textbf{Verify:}
\begin{lstlisting}[language=bash]
docker exec -it notes-db psql -U notes -d notesdb -c "SELECT 1"
\end{lstlisting}
\end{praticabox}

## Database
- RDBMS: PostgreSQL 16
- ORM: Prisma 6.x
- Connection string: in .env as DATABASE_URL
- Schema: prisma/schema.prisma

## Database Schema (Prisma)
model Note {
  id        String   @id @default(uuid())
  title     String   @db.VarChar(200)
  content   String   @db.Text
  tags      String[] @default([])
  createdAt DateTime @default(now()) @map("created_at")
  updatedAt DateTime @updatedAt @map("updated_at")
  @@map("notes")
}

## Database Constraints
- NEVER use raw SQL queries. ALWAYS use Prisma Client.
- NEVER put the connection string in the code. Use .env.
- Migrations MUST be generated with
  npx prisma migrate dev.
- The PrismaClient instance MUST be a singleton.
- Use @map to map JS camelCase -> DB snake_case.
\end{lstlisting}

\section{Introduction to MCP: AI Connected to the Database}

This is where the \textbf{Model Context Protocol (MCP)} comes into play --- the system that allows AI to interact directly with external tools such as databases, file systems, and APIs.

\subsection{What Is MCP in 30 Seconds}

Normally, when you ask the AI ``show me the database tables'', it has no access to the database. It can only imagine the answer. With MCP, the AI can \textbf{actually} query the database, read the real results, and use them to generate more accurate code.

\begin{figure}[H]
\centering
\includegraphics[width=0.75\textwidth]{03-mcp-confronto}
\caption{Without MCP vs.\ With MCP}
\end{figure}

\begin{praticabox}[title={Hands-On --- Configure the PostgreSQL MCP Server (optional)}]
Create the file \texttt{.vscode/mcp.json} in your project:
\begin{lstlisting}[language=json]
{
  "servers": {
    "postgres": {
      "command": "npx",
      "args": ["-y",
               "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL":
          "postgresql://notes:notes123@localhost:5432/notesdb"
      }
    }
  }
}
\end{lstlisting}
\end{praticabox}

\begin{tipbox}
Even without MCP, the project works perfectly --- the AI simply will not be able to run live queries against the database.
\end{tipbox}

\begin{notebox}
MCP\index{MCP} is Anthropic's open standard for connecting AI models to external tools~\cite{anthropic:mcp,mcp:spec}. It works like USB-C: a universal protocol for any connection. MCP servers expose \textbf{Tools} (actions), \textbf{Resources} (read-only data), and \textbf{Prompts} (templates).
\end{notebox}

\section{Migration from In-Memory to PostgreSQL}

\begin{praticabox}[title={Hands-On --- Prisma installation and migration}]
In Copilot Agent Mode:
\begin{lstlisting}[language={}]
Re-read the updated _CONTEXT.md. The Notes API project must
migrate from in-memory storage to PostgreSQL with Prisma.

Proceed as follows:
1. Install Prisma as a dependency
2. Initialize Prisma with npx prisma init
3. Create the Prisma schema according to _CONTEXT.md
4. Create the .env file with DATABASE_URL
5. Generate the initial migration
6. Create src/lib/prisma.js (singleton PrismaClient)
7. Rewrite src/services/notesService.js to use Prisma
8. Update the tests to use a test database
9. Verify that all endpoints work
\end{lstlisting}
\end{praticabox}

\subsection*{What Changes in the Code}

\textbf{Before (in-memory):}
\begin{lstlisting}[language=JavaScript]
// notesService.js - old version
let notes = [];

export function getAllNotes() {
  return notes;
}
\end{lstlisting}

\textbf{After (PostgreSQL):}
\begin{lstlisting}[language=JavaScript]
// notesService.js - new version
import prisma from '../lib/prisma.js';

export async function getAllNotes() {
  return await prisma.note.findMany({
    orderBy: { createdAt: 'desc' }
  });
}
\end{lstlisting}

\begin{quote}
The change is isolated in the service layer. The controllers, routes, and middleware remain identical. This is the advantage of the layered architecture defined in the \texttt{\_CONTEXT.md}.
\end{quote}

\section{Verification and Testing}

\begin{praticabox}[title={Hands-On --- Database testing}]
\begin{lstlisting}[language=bash]
# Start the server
npm run dev

# Create a note (now saved in the database!)
curl -X POST http://localhost:3000/api/notes \
  -H "Content-Type: application/json" \
  -d '{"title": "Database note",
       "content": "This one is persistent!",
       "tags": ["postgres"]}'

# Restart the server and verify that the note is still there
npm run dev
curl http://localhost:3000/api/notes
\end{lstlisting}

\end{praticabox}
