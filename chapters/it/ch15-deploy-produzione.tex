\chapter{Deploy in Produzione}\index{deploy}\index{CI/CD}

\section*{Cosa Costruirai}
L'intero stack deployato su piattaforme cloud:
\begin{itemize}
  \item Database PostgreSQL su \textbf{Neon} (serverless, gratuito)
  \item Backend Express.js su \textbf{Railway} (PaaS semplice)
  \item Frontend React su \textbf{Vercel} (ottimizzato per frontend)
  \item App mobile pubblicata sugli store (Cap.~13)
  \item Dominio personalizzato e HTTPS
\end{itemize}

\textbf{Tempo stimato}: 60--90 minuti\\
\textbf{Prerequisito}: Applicazione testata (Cap.~14)

\section{Database: Neon}

\begin{praticabox}[title={Pratica --- Setup Neon}]
\begin{enumerate}
  \item Vai su \texttt{neon.tech} e registrati
  \item Crea un nuovo progetto: ``notes-app''
  \item Copia la \textbf{connection string} (formato: \texttt{postgresql://user:pass@host/db?sslmode=require})
\end{enumerate}
\end{praticabox}

\begin{praticabox}[title={Pratica --- Migra lo schema}]
\begin{lstlisting}[language=bash]
cd backend

# Aggiorna .env con la connection string di Neon
# DATABASE_URL=postgresql://user:pass@ep-xxx.neon.tech/neondb

# Esegui le migrazioni sul database di produzione
npx prisma migrate deploy

# Verifica
npx prisma studio
\end{lstlisting}
\end{praticabox}

\begin{warnbox}
Usa \texttt{prisma migrate deploy} in produzione, \textbf{mai} \texttt{prisma migrate dev}. Il comando \texttt{dev} è interattivo e può cancellare dati. Il comando \texttt{deploy} applica solo le migrazioni pendenti.
\end{warnbox}

\section{Backend: Railway}\index{Railway}

\begin{praticabox}[title={Pratica --- Setup Railway}]
\begin{enumerate}
  \item Vai su \texttt{railway.app} e registrati con GitHub
  \item Clicca ``New Project'' $\rightarrow$ ``Deploy from GitHub repo''
  \item Seleziona il repository \texttt{notes-fullstack}
  \item Railway rileva automaticamente che è un progetto Node.js
\end{enumerate}
\end{praticabox}

\subsection{Variabili d'ambiente}

\begin{center}
\begin{small}
\begin{tabularx}{\textwidth}{l X}
\toprule
\textbf{Variabile} & \textbf{Valore} \\
\midrule
\texttt{DATABASE\_URL} & Connection string Neon \\
\texttt{JWT\_SECRET} & Stringa casuale $\geq$ 64 caratteri \\
\texttt{JWT\_EXPIRES\_IN} & 1h \\
\texttt{REFRESH\_TOKEN\_EXPIRES\_IN} & 7d \\
\texttt{GOOGLE\_CLIENT\_ID} & Il tuo client ID Google \\
\texttt{GOOGLE\_CLIENT\_SECRET} & Il tuo client secret Google \\
\texttt{GITHUB\_CLIENT\_ID} & Il tuo client ID GitHub \\
\texttt{GITHUB\_CLIENT\_SECRET} & Il tuo client secret GitHub \\
\texttt{FRONTEND\_URL} & https://notes-app.vercel.app \\
\texttt{NODE\_ENV} & production \\
\bottomrule
\end{tabularx}
\end{small}
\end{center}

\begin{praticabox}[title={Pratica --- Configura il build}]
\begin{lstlisting}[language={}]
Configura il backend per il deploy su Railway:
1. Aggiungi "start": "node src/index.js" in package.json
2. Specifica la root directory come "backend/"
3. Il server deve ascoltare sulla porta dalla variabile PORT
4. Aggiungi health check: GET /api/health -> { status: "ok" }
\end{lstlisting}
\end{praticabox}

\subsection{Deploy e verifica}

\begin{lstlisting}[language=bash]
git push origin main
# Railway rileva il push e fa il deploy automaticamente

# Verifica (dopo 1-2 minuti)
curl https://notes-api-production.up.railway.app/api/health
# {"status":"ok"}
\end{lstlisting}

\section{Frontend: Vercel}\index{Vercel}

\begin{praticabox}[title={Pratica --- Setup Vercel}]
\begin{enumerate}
  \item Vai su \texttt{vercel.com} e registrati con GitHub
  \item Clicca ``Add New Project'' $\rightarrow$ Seleziona il repository
  \item Configura:
  \begin{itemize}
    \item \textbf{Root Directory}: \texttt{frontend}
    \item \textbf{Framework Preset}: Vite
    \item \textbf{Build Command}: \texttt{npm run build}
    \item \textbf{Output Directory}: \texttt{dist}
  \end{itemize}
\end{enumerate}
\end{praticabox}

Variabile d'ambiente:
\begin{center}
\begin{tabularx}{\textwidth}{l X}
\toprule
\textbf{Variabile} & \textbf{Valore} \\
\midrule
\texttt{VITE\_API\_URL} & \texttt{https://notes-api-production.up.railway.app/api} \\
\bottomrule
\end{tabularx}
\end{center}

\begin{praticabox}[title={Pratica --- Aggiorna CORS e OAuth}]
\begin{lstlisting}[language={}]
Dopo il deploy, aggiorna:
1. Backend (Railway): aggiungi dominio Vercel a FRONTEND_URL
2. Google Cloud Console: aggiungi dominio Vercel agli URI
   di redirect autorizzati
3. GitHub OAuth Settings: aggiungi dominio Vercel al callback

Aggiorna CORS per accettare richieste da:
- http://localhost:5173 (sviluppo)
- https://notes-app.vercel.app (produzione)
\end{lstlisting}
\end{praticabox}

\section{Dominio Personalizzato}

\textbf{Vercel (Frontend):}
\begin{enumerate}
  \item Settings $\rightarrow$ Domains $\rightarrow$ Add
  \item Inserisci il dominio (es.\ \texttt{notes.tuodominio.com})
  \item Configura DNS con CNAME verso \texttt{cname.vercel-dns.com}
\end{enumerate}

\textbf{Railway (Backend):}
\begin{enumerate}
  \item Settings $\rightarrow$ Domains $\rightarrow$ Add Custom Domain
  \item Inserisci il sottodominio (es.\ \texttt{api.notes.tuodominio.com})
  \item Configura DNS con il CNAME fornito da Railway
\end{enumerate}

HTTPS è automatico su entrambe le piattaforme.

\section{Monitoraggio}

\begin{praticabox}[title={Pratica --- Logging di produzione}]
\begin{lstlisting}[language={}]
Aggiungi logging strutturato al backend:
1. Installa pino (logger JSON veloce per Node.js)
2. Logga ogni richiesta: metodo, path, status code, tempo
3. Logga errori con stack trace (solo nei log, MAI nelle risposte)
4. In sviluppo: output leggibile. In produzione: JSON
5. Escludi dati sensibili dai log (token, password, cookie)
\end{lstlisting}
\end{praticabox}

\begin{praticabox}[title={Pratica --- Health check}]
\begin{lstlisting}[language={}]
Crea un endpoint /api/health che restituisce:
{
  "status": "ok",
  "version": "1.0.0",
  "uptime": 12345,
  "database": "connected"
}

L'endpoint verifica la connessione al database con una
query Prisma semplice (SELECT 1).
\end{lstlisting}
\end{praticabox}

Usa un servizio gratuito come UptimeRobot per monitorare il health check ogni 5~minuti e ricevere notifiche se il backend va giù.
