\chapter{Progetto 4: Database e Modello Dati con PostgreSQL}\index{PostgreSQL}\index{Prisma}

\section*{Cosa Costruirai}
In questo capitolo trasformerai la Notes API del Capitolo~6 da storage in-memory a un \textbf{backend con database reale}:
\begin{itemize}
  \item PostgreSQL come database relazionale
  \item Prisma come ORM (Object-Relational Mapper)
  \item Migrazioni automatiche per la gestione dello schema
  \item Primo contatto con MCP (Model Context Protocol) per l'integrazione database
\end{itemize}

\textbf{Tempo stimato}: 45--60 minuti\\
\textbf{Prerequisito}: Progetto \texttt{notes-api} del Capitolo~6

\begin{notebox}
\textbf{Box Teoria --- Database, ORM e Migrazioni.} In questo capitolo passerai da un semplice array in memoria a un \textbf{database relazionale} reale (PostgreSQL). I dati vengono organizzati in \textbf{tabelle} con colonne tipizzate (testo, numeri, date). Un \textbf{ORM} (Object-Relational Mapper) come Prisma traduce le operazioni nel linguaggio del database (SQL) al posto tuo: scrivi \texttt{prisma.note.create(\{title: "Ciao"\})} e Prisma genera l'SQL corrispondente. Le \textbf{migrazioni} sono script che modificano la struttura del database in modo tracciabile --- come un changelog per le tabelle. L'IA genererà tutto questo; il tuo compito è verificare che lo schema dei dati abbia senso.
\end{notebox}

\begin{notebox}
\textbf{Box Tooling --- Stack scelto per questo esempio.}
\begin{itemize}
  \item \textbf{Database:} PostgreSQL~16
  \item \textbf{ORM:} Prisma~6.x
  \item \textbf{Migrazioni:} Prisma Migrate (integrate)
\end{itemize}
\textbf{Alternative equivalenti:} MySQL/MariaDB con TypeORM, Python con SQLAlchemy, Go con GORM, MongoDB con Mongoose. Il \textbf{pattern} (database relazionale + ORM + migrazioni tracciabili) è universale. Se avessi scelto Drizzle al posto di Prisma, o SQLAlchemy al posto di entrambi, il metodo 0-code (Context Engineering + ADLC) sarebbe stato identico.
\end{notebox}

\section{Installare PostgreSQL}

\begin{praticabox}[title={Pratica --- Setup PostgreSQL}]
\textbf{Opzione A --- Docker (raccomandata):}
\begin{lstlisting}[language=bash]
docker run --name notes-db \
  -e POSTGRES_USER=notes \
  -e POSTGRES_PASSWORD=notes123 \
  -e POSTGRES_DB=notesdb \
  -p 5432:5432 -d postgres:16
\end{lstlisting}

\textbf{Opzione B --- Installazione locale (Windows):}
Scarica da \url{https://www.postgresql.org/download/windows/}, installa con pgAdmin, crea un database \texttt{notesdb}.

\textbf{Opzione C --- Cloud (Supabase/Neon):}
Crea un progetto gratuito su \url{https://supabase.com} o \url{https://neon.tech} e copia la connection string.

\textbf{Verifica:}
\begin{lstlisting}[language=bash]
docker exec -it notes-db psql -U notes -d notesdb -c "SELECT 1"
\end{lstlisting}
\end{praticabox}

\section{Introduzione a MCP: L'IA Collegata al Database}

Qui entra in gioco il \textbf{Model Context Protocol (MCP)} --- il sistema che permette all'IA di interagire direttamente con strumenti esterni come database, file system e API.

\subsection{Cos'è MCP in 30 secondi}

Normalmente, quando chiedi all'IA ``mostrami le tabelle del database'', lei non ha accesso al database. Può solo immaginare la risposta. Con MCP, l'IA può \textbf{effettivamente} interrogare il database, leggere i risultati reali e usarli per generare codice più accurato.

\begin{figure}[H]
\centering
\includegraphics[width=0.75\textwidth]{03-mcp-confronto}
\caption{Senza MCP vs.\ Con MCP}
\end{figure}

\begin{praticabox}[title={Pratica --- Configurare MCP Server PostgreSQL (opzionale)}]
Crea il file \texttt{.vscode/mcp.json} nel tuo progetto:
\begin{lstlisting}[language=json]
{
  "servers": {
    "postgres": {
      "command": "npx",
      "args": ["-y",
               "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL":
          "postgresql://notes:notes123@localhost:5432/notesdb"
      }
    }
  }
}
\end{lstlisting}
\end{praticabox}

\begin{tipbox}
Anche senza MCP, il progetto funziona perfettamente --- l'IA semplicemente non potrà fare query live al database.
\end{tipbox}

\begin{notebox}
MCP\index{MCP} è lo standard aperto di Anthropic per connettere modelli IA a strumenti esterni~\cite{anthropic:mcp,mcp:spec}. Funziona come USB-C: un protocollo universale per qualsiasi connessione. I server MCP espongono \textbf{Tools} (azioni), \textbf{Resources} (dati in sola lettura) e \textbf{Prompts} (template).
\end{notebox}

\section{Migrazione da In-Memory a PostgreSQL}

\begin{praticabox}[title={Pratica --- Installazione Prisma e migrazione}]
In Copilot Agent Mode:
\begin{lstlisting}[language={}]
Rileggi il _CONTEXT.md aggiornato. Il progetto Notes API deve
migrare da storage in-memory a PostgreSQL con Prisma.

Procedi cosi':
1. Installa Prisma come dipendenza
2. Inizializza Prisma con npx prisma init
3. Crea lo schema Prisma secondo il _CONTEXT.md
4. Crea il file .env con DATABASE_URL
5. Genera la migrazione iniziale
6. Crea src/lib/prisma.js (singleton PrismaClient)
7. Riscrivi src/services/notesService.js per usare Prisma
8. Aggiorna i test per usare un database di test
9. Verifica che tutti gli endpoint funzionino
\end{lstlisting}
\end{praticabox}

\subsection*{Cosa cambia nel codice}

\textbf{Prima (in-memory):}
\begin{lstlisting}[language=JavaScript]
// notesService.js - vecchia versione
let notes = [];

export function getAllNotes() {
  return notes;
}
\end{lstlisting}

\textbf{Dopo (PostgreSQL):}
\begin{lstlisting}[language=JavaScript]
// notesService.js - nuova versione
import prisma from '../lib/prisma.js';

export async function getAllNotes() {
  return await prisma.note.findMany({
    orderBy: { createdAt: 'desc' }
  });
}
\end{lstlisting}

\begin{quote}
Il cambio è isolato nel service layer. I controller, le route e il middleware restano identici. Questo è il vantaggio dell'architettura a layer definita nel \texttt{\_CONTEXT.md}.
\end{quote}

\section{Verifica e Test}

\begin{praticabox}[title={Pratica --- Test del database}]
\begin{lstlisting}[language=bash]
# Avvia il server
npm run dev

# Crea una nota (ora salvata nel database!)
curl -X POST http://localhost:3000/api/notes \
  -H "Content-Type: application/json" \
  -d '{"title": "Nota nel database",
       "content": "Questa e` persistente!",
       "tags": ["postgres"]}'

# Riavvia il server e verifica che la nota sia ancora li'
npm run dev
curl http://localhost:3000/api/notes
\end{lstlisting}

\end{praticabox}
